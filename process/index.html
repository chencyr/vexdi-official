從第一封信到上線後一年，
每一步都有可驗收的東西。

我們把一個專案拆成五個階段。每階段時程是固定的、產出是具體的、收費是分段的 —— 你隨時都可以看到「現在在做什麼、為什麼花這個錢、下週會看到什麼」。

00 · DAY 0

30 分鐘線上聊聊（免費）

你寫信來，我隔天回。約 30 分鐘 Google Meet。 目的不是賣案子，是確認這件事「值不值得做、由我做合不合適」—— 有時候你需要的可能是一個 freelancer，或者根本不是現在做。

聊完我會用 1–2 天寫一份「初步建議書」給你： 這件事我覺得該做 / 該緩 / 該用別的方法、可能的範圍與大致預算。 這份建議書是免費的，拿走不簽約沒有任何成本。

01 · DISCOVERY · 1–2 週

實地訪談 + 流程拆解

確認要合作後，第一階段是把你業務裡的真實流程搞清楚。 不是「需求書」、不是「規格表」，是實際看你和你員工怎麼做事的。

  • 1–2 次上門（大台北）或視訊（外縣市），各 2–3 小時，看現場、聽抱怨。
  • 跟核心使用者做一輪深訪。SaaS 案子多訪一線員工，網站案子多訪行銷與業務。
  • 整理出問題清單與優先序，寫成「現況地圖」文件。

結尾交付：範圍書（要做什麼、不做什麼）+ 正式報價（分階段付款）+ 時程表。 這份範圍書是雙方接下來的合約附件 —— 後面任何爭議都回到這份文件談。

02 · DESIGN · 2–4 週

設計稿 + 互動原型

範圍書簽完，進入設計。我們不會「先寫程式再說」—— 這個階段把所有畫面、所有流程、所有邊界情境想清楚，再開始寫 code。

  • 低保真 wireframe（流程對不對） → 高保真 UI（細節怎麼長）。
  • 可點擊原型：Figma prototype，所有主要 user flow 你可以親自跑一遍。
  • 2 輪修改包含在報價內。第 3 輪以後若有重大方向改變，會討論是否影響範圍書。

設計階段結尾你會親自簽署「設計確認書」。簽了之後就進入 BUILD —— 程式碼開始大量寫， 這個時候才回頭改設計就會貴。所以這個簽名很重要。

03 · BUILD · 4–12 週

每週可預覽版本

開始寫程式碼。我們以週為單位推進 —— 每週末你會收到一封「本週進度信」，裡面有：本週做了什麼、可預覽連結、下週要做什麼、有沒有需要你決定的事。

  • 每週可預覽版本（部署到 staging 環境，你隨時點開看）。
  • 所有程式碼在 GitHub repo，從第一天就有，你可以全程看版本歷史。
  • 每兩週一次 30 分鐘 sync call，聊上週進度、確認下週方向。
  • 意外狀況 / 範圍變更 / 卡點都會立刻說，不會等到驗收日才告訴你。

BUILD 中段 + 結尾各收一次款（範圍書會寫清楚比例）。 標準分段：30% / 30% / 30% / 10%（簽約 / BUILD 中 / 上線 / 保固結束）。

04 · HANDOVER · 1 週

上線 + 原始碼 + 文件交付

上線當天，我們把所有東西交給你 —— 不是「等你之後找誰」、不是「我們託管在我們平台」， 是你拿走後可以自己跑、自己改、找別人接手都不會被綁住。

  • GitHub repository 轉移到你的組織帳號（你擁有所有 commit history）。

WHY · 為什麼這樣做

每一條都是過去看過誰被坑了，反過來定的。

為什麼有「免費 30 分鐘」？

太多客戶被「先簽 NDA、付諮詢費」的工作室拖走 5 萬塊只拿到一份 PPT。前 30 分鐘聊不出價值的工作室，後面也聊不出。

為什麼有「範圍書」？

大部分外包糾紛都來自「我以為包這個 / 我以為不包」。簽前寫清楚，比簽後吵清楚便宜很多。

為什麼「每週可預覽」？

三個月後驗收才打開，發現方向偏了 —— 來不及。每週看，偏一點點就能轉，不會偏一里。

為什麼「智財 100% 歸你」？

不能讓客戶被工作室綁住。我們得靠下一個案子賺錢，不是靠把這個案子的原始碼留在手上勒索。

為什麼分段付款？

單方面預付全款 —— 工作室沒誘因準時。一次付清 —— 客戶沒有風險控制。分段，雙方都站在同一邊。

為什麼「2 小時 workshop」？

沒有交接的交付會變成「半年後客戶又來找你」。前面把他教會，他自由，你也自由。
